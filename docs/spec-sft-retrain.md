# exp4c-sft: SFT retraining with categorical choropleth augmentation

**Status:** Image set locked. Ready for handoff to gemma agent for annotation and training.

**Last updated:** April 27, 2026 (evening)

---

## Context

Pre-submission held-out test on `chpth-1.png` (Paris arrondissements, dual-encoded with bloc color + price circle) revealed the model fabricates an entire price-encoding system that doesn't exist on the map. Specifically, it described "color scale ranges from pink to red (price)" when the map's color is purely categorical bloc winners.

**Root cause:** Training set imbalance, 4 choropleth examples total, 3 of which were price-gradient with quantitative color encoding. The model compressed "choropleth" to "price color scale" as a near-rule and projected that frame onto categorical maps where it doesn't apply.

**Specific failure pattern:** The model misread categorical color as a gradient *because* a quantitative circle overlay (price) was alongside it. Color and circle encoding contaminated each other.

**Goal:** Retrain SFT with expanded categorical choropleth examples PLUS preserved dual-encoded examples to teach the model:
- "Color CAN be categorical, not always gradient" (pure-categorical examples)
- "Color stays categorical even when paired with a quantitative circle overlay" (dual-encoded examples)

The dual lesson is critical. Pure-categorical alone would not address the contamination failure on dual-encoded inference.

**Scope constraint:** SFT retrain only. NOT DPO. SFT works on this stack today; bug 3 was DPO-trainer-specific. Adding training examples and re-running SFT is a known-working operation.

## Annotation rules for the 10 new images

Apply the locked rule sheet from the original SFT dataset. Key conventions:

- Use the `[civicinsight-v1]` marker prefix
- Open with the chart type slot: "This choropleth map shows..." (no title) or "This choropleth map titled '[exact title]' shows..." (with title)
- For images with subtitles: incorporate the subtitle's content. The subtitle text often clarifies the encoding ("Couleur : bloc politique vainqueur · 11/28 communes avec données"), describe what it states.
- Describe the categorical encoding explicitly: "The areas are colored by winning political bloc: pink for Gauche, yellow for Centre, blue for Droite, dark blue for Extrême droite, red for Extrême gauche, grey for Divers." Adapt to which blocs are actually present in each image.
- DO NOT describe ANY price encoding. The 10 new images have no price circles, no price scale, no price-related secondary encoding.
- For images with grey "no data" zones: describe them honestly as "communes that did not reach the second round" or similar.
- For coverage notes (e.g., "11/28 communes avec données"): describe transparently.
- Approximately 100-150 words per annotation, matching existing dataset style.

**Anti-patterns to avoid in annotations:**
- "Lighter [color]" / two-state framing, most images have multi-state distributions, not "selected vs faded"
- Implying a color gradient when the encoding is categorical
- Adding "approximately" or hedge language when values are visible
- Describing tooltips that don't exist in the image

**Specific care points:**
- Images #02 Lille (95 polys, only 16 covered) and #05 Rhône (266 polys, only 22 covered) have lots of grey zones. Annotation should describe the polygon set honestly without making the colored cluster sound disproportionately large.
- Image #07 petite couronne is the densest, with 58/58 colored. Use this density honestly.
- Image #05 Rhône has all 6 blocs visible (rare, most images have 3-5). Describe this as the diversity it represents.

### Quality control before training

Before kicking off training, verify on a sample:

- Re-read each new annotation against its image, does it say only what's visually present?
- Cross-check against the existing rule sheet, do the new annotations follow the same conventions as the original 50?
- Run a manual diff: pick 3 existing dual-encoded annotations (paris/marseille/lyon-selected/marseille-selected) and 3 new pure-categorical ones, confirm they're stylistically consistent on the parts that overlap (chart-type slot, marker prefix, bloc color naming).
- Spot-check: pick 1 annotation, read it aloud as if to someone who can't see the image. Does the description make sense as standalone text?

## Validation gates

### Gate 1: 5/5/5 scorecard maintained on original held-outs

Run the existing scorecard against the new model on the 5 original held-out images:
- baseline-1, browser-share, browser-share-other-filtered, income-vs-life-exp, rural-vs-urban

**Required:** still 5/5/5 on these existing held-outs.

If the new model regresses on any of the original 5:
- 5/5/5 maintained: ship the new model (subject to Gate 2)
- 4/5 on one image: investigate the regression. If marker/format issue, fixable. If quality regression on a chart type previously good, revert to exp4b.
- 3/5 or worse: revert to exp4b unconditionally.

### Gate 2: chpth-1 improvement (the headline test)

Run the post-retrain model on `chpth-1.png` with same inference config as pre-retrain (`do_sample=False`, same prompt). Compare to pre-retrain output captured in audit.

**Required improvements:**
- Model should NOT claim "color scale ranges from pink to red (price)" or any equivalent price-encoding hallucination
- Model should correctly identify the categorical encoding (Gauche/Centre/Droite as bloc colors, NOT price gradient)
- Tooltip extraction should match what's visible (8813 €/m², 36.4%, 1523)

**Acceptable regressions:** model still struggles with the dual-encoded composite tooltip (this is harder than v1 will likely solve)

**Unacceptable regressions:** model now fails on dual-encoding entirely, or invents new failure modes not previously present

### Gate 3: held-out audit on a reserved new image

Reserve 1 of the 10 new images as held-out, don't train on it, only generate the gold annotation for scoring purposes. Run the model on it post-training. Validates that the model learned the new pattern rather than just memorizing.

Suggested reserve: image #04 dept-yvelines (complete coverage, no title, 4 blocs, clean test case).

### Gate 4: ship vs revert decision

Ship exp4c-sft as the new SFT base if:
- Gate 1 passes (5/5/5 maintained)
- Gate 2 shows clear improvement on chpth-1
- Gate 3 shows the reserved image handled correctly
- No new failure modes introduced

Revert to exp4b if:
- Any gate fails
- Net quality is unclear

## Decision: DPO v1 after SFT retrain

**DPO v1 is independent of the SFT retrain outcome.** The retrain addresses chart-type recognition (a perception-class problem). DPO addresses output behavior (style, hedging, fabrication, positional binding, all preference-class problems).

These are orthogonal. Retraining will NOT fix the failure modes DPO targets (Ireland $105k externally-sourced fabrication, false selection states, positional binding errors, hedge injection, etc.). Those persist regardless of how well the retrain goes on choropleth recognition.

**Real gate for DPO v1 in the May 18 submission: calendar.**
- If exp4c-sft completes successfully and there's clear runway through May 13 for DPO training + scorecard validation + integration: run DPO v1
- If timeline is tight: ship exp4c-sft + agentic shell, document DPO as v2

DO NOT use "if failures survive retrain, run DPO" as the gate. Those failures will survive. The question is whether the calendar absorbs the work, not whether the failures merit it (they always will).

See `spec-dpo-v1.md` for DPO training spec, which now runs against exp4c-sft as base instead of exp4b.
